# Repo Difficulty Benchmark Designer

使用此 skill 从 `E:\claudeCoding\repos` 下的仓库中设计 benchmark 任务，用于 qwen 和 opus 之间的对比评测。

## 何时调用

当用户要求以下操作时调用此 skill：

- 从 `E:\claudeCoding\repos` 中的仓库创建任务或 prompt
- 为 qwen 和 opus 设计 benchmark 问题
- 创建旨在拉大两个模型得分差距的高难度评测任务
- 准备后续用于 `prompt.txt` 或评测流程的基于仓库的 prompt

不要将此 skill 用于通用代码解释或普通功能实现。

## 参数说明

调用格式：`Use Skill: repo-difficulty-benchmark-designer <仓库名> [任务类型]`

支持的任务类型：

- **bug修复**（默认）：修复已存在功能中的正确性问题
- **功能优化**：优化现有功能的性能、可读性、可维护性等
- **功能迭代**：在现有架构基础上添加新功能或扩展能力
- **受限重构**：在保留外部接口的前提下重构内部实现

如果未指定任务类型，默认使用 `bug修复`。

## 核心目标

你的目标是设计能最大化 opus 在同一仓库任务上得分高于 qwen 的概率的任务。

不要承诺必然的排序结果。而是明确优化以下方面：

- 更高概率让 opus 表现更好
- 更低概率让 qwen 通过浅层局部推理解决好任务
- 最终质量评分中更强的区分度

## 交付目标对齐

此 skill 必须与评测侧的交付约束保持一致。

设计的 prompt 应让合理的配对评测能够满足以下所有条件：

- qwen 通过率 **< 0.7**
- claude 通过率 **> qwen**
- 相对差距 **(claude - qwen) / qwen > 20%**

这意味着任务不能仅仅允许产生一个赢家；它应在共享评分标准下自然地支撑一个**清晰且有证据支撑的差距**。

避免以下类型的任务：

- 浅层部分修复看起来仍然接近完成
- 页面级别的补丁就能获取大部分可用分数
- 两个模型很可能落在同一分数段，仅有微小差异
- 任务过于模糊，导致评分标准后续需要做人为的分离工作

## 语言和语气默认值

除非用户明确要求其他语言或风格：

- 使用**中文**撰写 benchmark prompt
- 语气像**一个开发者在 Slack / 飞书 / issue 里跟同事说活儿**，不是写文档也不是写 spec
- 口语化、直接、有细节。读起来像真有人在描述 TA 遇到的生产问题
- 纯文本就行，别搞 Markdown 排版
- 保持紧凑，能直接粘进 Claude Code 不截断

好 prompt 读起来像：

- "线上有个 bug，我跟你同步一下..." — 然后开始描述症状
- "上次那个重构留了个坑..." — 然后解释因果关系
- "用户反馈 xx 不对，我排查了一下应该是..." — 给出排查链路
- "之前改 xx 的时候忘了 yy，导致..." — 带因果链的跟进
- "新功能跑了几周，发现一个边界 case..." — 生产发现

不要弄成：

- 带编号章节的文档体（"### 1. 背景"、"## 问题描述"）
- 教科书式的陈述
- "请完成以下任务"、"你的职责是"
- 任何一眼就能看出是 AI prompt 的东西
- 暴露这是评测框架的元语言

prompt 不应该像一份需求文档。它应该像一段真实的对话：有事说事，逻辑清晰，但语气就是你跟同事说话的语气。

## 输出紧凑性默认值

除非用户明确要求更长的 prompt：

- 输出 `prompt.txt`（不是 .md）
- 尽量控制在 **一段到十几行** 之内
- 不用 Markdown 标题，不用编号列表，不用代码块
- 信息密度高但不啰嗦
- 只保留对实现有实质约束的细节

如果压缩后实在讲不清楚，可以多写几句。但永远不要为了"格式完整"去加废话。

## 设计理念

倾向于奖励全局推理、架构判断、精确性和有纪律的验证的任务。

任务应让薄弱的局部启发式方法失败。优先选择要求模型完成以下工作的任务：

- 跨多个文件或层级连接行为
- 从现有代码模式中推断隐藏的不变量
- 在修改内部实现时保留架构和公共契约
- 诊断微妙的正确性问题，而非做明显的编辑
- 在多个竞争的实现策略中做出选择并论证选择理由
- 以符合仓库惯例的方式添加或更新验证
- 在代码修改后清晰地解释权衡

避免容易被局部编辑暴力解决的任务，例如：

- 症状明显的孤立单文件 bug 修复
- 搜索替换式的重命名
- 简单的 CRUD 新增
- 有明确指令的直接语法迁移
- 不需要阅读更广泛的仓库上下文就能判断成功的任务

## 各任务类型的设计策略

根据指定的任务类型，调整 prompt 设计的侧重点。不同类型有不同的"浅层修复全是坑"方式。

### bug修复

定位问题 → 理解根因 → 跨层修复 → 防止回归。这是默认类型，当前设计理念和分离经验教训主要针对此类型。

浅层修复的典型缺陷：
- 只在症状表现层加 guard，不动数据源头
- 修了 A 调用点，漏了同样调用链上的 B、C
- 处理了正常路径，退化数据下仍然出错

### 功能优化

不是"代码写错了"，而是"代码能跑但不够好"。坑藏在"优化不能破坏正确性"这条线上。

高价值的优化任务：
- 先理解现有正确性保证（隐藏的不变量、隐含的时序依赖、并发语义），再动手
- 优化后在多维度上都不退化（如：提高了时间效率但内存暴涨不算赢）
- 需要实际测量验证，而不是凭感觉改

浅层修复的典型缺陷：
- 为了优化改掉了隐式保证（如：把同步调用改异步，但调用方依赖了同步语义）
- 局部优化引入了全局退化（如：缓存加得太激进，数据新鲜度出问题）
- 只改热点代码，不改调用方，优化收益被调用方稀释

好的功能优化 prompt 应该自然包含：
- 一个现状描述（"这个接口现在太慢了"、"频繁 GC"）
- 隐含的约束（"调用方不接受异步"、"不能改 API 签名"）
- 隐含的验证要求（"优化之后 QPS 不能低于原来"）

### 功能迭代

在骨架上长新肉。任务不是修东西，而是在现有约定和抽象上扩展。坑藏在"新增的东西和旧的东西要保持一致"。

高价值的功能迭代任务：
- 需要读懂现有抽象、约定、不变量，然后在新路径上遵守同样的规则
- 既有代码路径不能退化（回归压力大）
- 如果在多个类似模块上做相同的事情，模式一致性很重要

浅层修复的典型缺陷：
- 在某个文件里新加了一段代码，但没继承周围的错误处理、日志、权限检查模式
- 复制粘贴了类似模块的代码，但没适配新模块的上下文
- 加了新功能，但把老的边界行为覆盖掉了

好的功能迭代 prompt 应该自然包含：
- "咱们现在有个 xx 模块，需求是要加一个类似 yy 的功能"
- "参考已有的 zz 实现来做，保持一致的错误处理和日志风格"
- "老的流程不能受影响"

### 受限重构

接口不能变，内部随便改。坑藏在"你怎么证明重构之后行为一样"。

高价值的受限重构任务：
- 重构需要触及多个内部实现文件
- 公共接口、协议格式、文件格式等边界不能动
- 需要用仓库现有的测试来证明等价性

浅层修复的典型缺陷：
- 改动了看似内部的细节，但实际上影响了外部行为
- 重构了一个类但没改它的所有创建点，留下了旧实现残余

好的受限重构 prompt 应该自然包含：
- "这个模块内部太乱了，XX 和 YY 耦合在一起，每次改都要动两个地方，想拆开"
- "对外接口不能变，测试不能挂"
- "重构完跑一下现有的测试确认没退化"

## 分离经验教训

将以下经验作为未来 prompt 设计的硬性指导。

### 1. 为什么早期的权限任务分离不够

权限 / 认证类的 prompt 看似很强，但当以下情况出现时，用浅层推理也能近似解决：

- 症状已经直接指向修复形式，例如"某页面越权可见"或"非创建者还能编辑/删除"
- 明显响应是在指定页面或操作中添加 redirect / guard / 创建者检查
- 大部分重要文件已经被症状本身点名
- 部分修复在评分中仍然看起来可接受，因为页面 guard + 一个服务端检查就覆盖了大部分可见症状
- 几乎没有隐藏的数据契约工作、查询行为、空状态处理或生命周期一致性需要推理

将这类任务视为**中风险分离任务**，而非优先默认任务。

### 2. 为什么后来的已删除用户数据链任务分离更好

当可见症状和真正修复被多层隔开时，出现了更强的任务：

- 外部触发器或生命周期事件
- 清理或变更逻辑
- 数据库引用一致性
- 空引用或悬空引用下的聚合 / 查询行为
- UI 渲染回退
- 未被第一个症状点名的二级页面

这种形态分离更好，因为浅层修复很可能停在一层，而更强的模型必须：

- 在与症状不同的模块中找到非显而易见的根因
- 同时修复数据清理和查询行为
- 在退化数据下保持公开浏览和历史展示稳定
- 推理可为空的运行时状态，而非仅处理正常路径类型
- 在修复后验证不止一条路径

### 3. 各任务类型的默认偏好顺序

在选择候选任务时，根据任务类型遵循对应的偏好顺序。以下按优先级从高到低排列。

#### bug修复偏好

- 删除 / 同步 / webhook / 重试 / 对账之后的数据生命周期回归
- 跨越 schema、查询和渲染层的引用一致性 bug
- 缺失关联导致静默丢数据的聚合或分页 bug
- 运行时数据实际状态与声明的类型 / 契约之间的状态漂移
- 一次修复必须同时保证公开页面、历史页面和管理页面正确的多界面回归

优先于主要归结为以下内容的任务：

- 添加权限检查
- 添加重定向
- 隐藏一个按钮
- 修补一个明显的服务端 action guard
- 在症状直接点名的文件中修改一个条件判断

#### 功能优化偏好

- 需要理解时序依赖或并发语义才能安全优化的性能热点
- 修改后需要在正确性和性能之间权衡的优化（如缓存策略、批处理粒度）
- 局部优化需要全局验证以防别处退化的场景
- 多语言或跨技术栈的同等优化（如：Java 后端和 Python 脚本都需要加速，且要保持一致）

优先于主要归结为以下内容的任务：

- 加个索引
- 调大或调小一个参数值
- 一眼能看出改什么且不需要验证

#### 功能迭代偏好

- 已有 N 个类似模块，需要新增第 N+1 个且必须遵守同样约定的场景
- 需要在现有抽象上扩展新能力（如：新增一种序列化格式、新增一个数据源适配器）
- 新功能与旧功能在同一个调用链上，一处改动影响多处
- 新增功能需要与现有错误处理、日志、监控体系对等

优先于主要归结为以下内容的任务：

- 复制粘贴一个文件改几下就能跑
- 新增无需考虑现有架构的独立端点

#### 受限重构偏好

- 多个内部类耦合严重，但公共接口不能变
- 重构需要调整多个 internal/private 实现，而公共测试是仅有的安全网
- 线程模型、IO 模型、生命周期管理发生实质性改变但 API 保持不动
- 需要从隐式依赖中提取显式接口而不破坏调用方

优先于主要归结为以下内容的任务：

- 重命名
- 搬一个方法的位置
- 加个参数

## 仓库选择工作流

### 1. 阅读仓库上下文

检视目标仓库以理解：

- 架构边界
- 主要包或模块
- 现有测试、脚本和验证机制
- 编码惯例
- 具有跨文件耦合的组件

### 2. 搜索强候选任务区域

根据指定的任务类型，侧重搜索不同属性的代码区域：

**通用属性（所有类型都适用）：**

- 多个交互模块
- 有状态的流程或生命周期钩子
- 需要保持一致性的错误处理路径
- 必须保持对齐的配置和运行时行为
- 揭示意图但不揭示确切实现的测试

**bug修复侧重：**

- 遥测、追踪、缓存、并发、schema 转换、适配器或协议处理
- 可能留下悬空数据的清理 / 删除 / webhook / 同步 / 重试流程
- 在缺失关联时行为不同的聚合、连接、分页、派生列表或回退渲染

**功能优化侧重：**

- 循环、热路径、缓存、批处理、IO 操作密集的代码
- 有隐藏的不变量或时序依赖、不能在优化中破坏的逻辑
- 存在 benchmark 或性能测试文件，可用于验证优化效果

**功能迭代侧重：**

- 已有多个类似实现的接口、抽象类、策略模式
- pipeline/filter/interceptor 等可插拔的扩展点
- 配置加载、序列化、协议解析等格式敏感的代码
- 存在可参考的同类模块实现

**受限重构侧重：**

- 职责混杂的大类、长方法、深度耦合的类簇
- 全局状态、单例、静态方法调用散布的模块
- 有公共测试但无单元测试的灰色地带代码

### 3. 选择有利于 opus 的任务形态

高价值任务形态根据任务类型有所不同：

**通用形态：**

- 需要推断不变量的部分实现补全
- 添加暴露隐藏正确性问题的验证或测试
- 将多个代码路径对齐到单一架构规则

**bug修复侧重：**

- 根因非显而易见的多文件 bug 诊断
- 修复证据分布在实现、测试和配置中的 bug
- 数据变更和下游读取方都必须修复的生命周期回归
- 必须在后端和 UI 之间一致处理运行时可空性、过期引用或部分数据的 bug

**功能优化侧重：**

- 修改必须在正确性和性能之间权衡，不可以只取其一
- 需要理解隐藏不变量才能安全优化的性能热点
- 多文件受同一优化影响，验证不能只测一处

**功能迭代侧重：**

- 必须保留外部行为的受限扩展
- 新功能与旧功能共享调用链，一处改动影响多处
- 新增代码必须遵循现有抽象和约定，不能另起炉灶

**受限重构侧重：**

- 必须保留外部行为的受限重构
- 改变内部实现但不允许改变公共契约
- 重构后的类需要在多个现有调用点正常工作

### 4. 增大分离压力

在撰写任务时，以保持现实感的方式增加难度：

- 要求触及多个相关文件
- 要求保留向后兼容性或接口形态
- 要求从现有模式中推理，而非发明新风格
- 要求验证，而非仅做代码编辑
- 要求简洁解释权衡或残余风险
- 保留一定模糊性让模型自己决策，但不能多到让任务不公平
- 确保至少一个关键文件未被症状明确点名
- 确保浅层单层修复会在查询、渲染、类型或清理行为中留下明显的隐藏缺口
- 确保部分修复在完整性、隐藏边界处理、退化数据行为、验证深度和最终交付质量上自然丢分

## 任务撰写规则

生成 prompt 时记住：**你不是在写文档，你是在跟同事说活儿**。

- 用中文，口语化。读起来像一个人跟另一个人说话
- 不要提 qwen 或 opus，不要暴露评分标准或 benchmark 策略
- 说清楚问题就行，不要过度指定——留空间让模型自己思考
- 要真实的代码修改，不只分析
- 仓库有测试机制的，让人跑测试验证
- 最后让人简单说一下改了啥、怎么确认的

一个 natural 的好 prompt 通常自然包含：

- 怎么发现问题的（"上线几周后发现..."、"用户反馈..."）
- 实际的症状和排查过程
- 跟仓库相关的具体约束（"别动 xx 接口"、"之前 yy 的流程不能断"）
- 隐含的跨文件正确性要求
- 自然地提到有哪些坑或要注意的点
- 至少一处浅层修复会明显漏掉的东西

风格上：

- 用 "我"、"咱们" 之类的人称，像真人在说话
- 可以是连续的段落，不用拆成小节
- 别用编号列表、Markdown 标题、代码块包装
- 文件名用自然的方式顺带提到，不用 `backtick` 也没关系

反例——绝对不要写成这样：

```
## 背景
本项目是一个...
## 问题描述
1. xx 功能存在缺陷
2. yy 与 zz 不一致
## 任务要求
请完成以下修改：...
```

正例——应该像这样：

bug修复风格的写法：
```
上次加的定时清理有个问题——它会把已经通知过的房源也删了。删完之后那个房源还在 kv.ee 上挂着的话，下次抓取又会当新房源抓进来、重新打分、重新发邮件通知。改的时候注意两点：1 别动现有的抓取和打分流程 2 已经存了的数据要能兼容。另外 API 那边展示的分数也得跟通知时一致——现在 building material 的加减分只在内存里算，没存库，API 那边重新算就对不上了。改完帮我跑一下测试。
```

功能优化风格的写法：
```
现在 report 导出当数据量上到 10w 行之后太慢了，一个接口能跑两分钟。我扫了一下，ExportService 那个循环里有个 O(n²) 的 join——没有索引命中，每次迭代都在扫全表。这个 join 逻辑是之前从 v2 迁移过来的，当时数据量小没暴露。优化方向是在循环外面先一次性查出来放 Map 里，循环里直接用。注意 ChunkedWriter 那边依赖了每次迭代的副作用，别踩坑。内存方面——数据量级参考现有的 maxRows 配置，别搞出 OOM。改完跑一下 ExportServiceBenchmark。
```

功能迭代风格的写法：
```
现有的 HTTP API 已经有了 token 认证和 basic auth 两种方式，现在需要再加一个 API key 认证。认证逻辑入口在 AuthInterceptor，现有的 Authenticator 接口两个实现分别是 TokenAuthenticator 和 BasicAuthenticator，你参考它们加一个 ApiKeyAuthenticator。api key 的格式是 "sk-xxx"，从 Header X-API-Key 里读。认证通过之后的行为和 token 认证一样——往 SecurityContext 里放一样的 principal。配置项加在 application.yml 里，key 叫 auth.api-key.enabled，默认 false。老的那两种认证方式不能受影响。
```

受限重构风格的写法：
```
DateRangeParser 这个类现在太大太乱了——里面把所有时区转换逻辑、格式兼容、fallback 策略全揉在一个方法里，一千多行。每次上游格式变一点都要改它，而且没有测试覆盖。想把它拆成几个独立的类：FormatDetector（识别输入格式）、Normalizer（不同格式统一到 ISO）、RangeResolver（算时间区间）。对外入口 DateRangeParser.parse() 这个 public 方法签名不能变，所有调用方不能有感知。拆完之后 DateRangeParser 只是组合这三个新类。现有的 DateRangeParserIntegrationTest 要全部通过。

## Prompt 压力检查清单

在最终确定任务之前，检查以下所有项。优先选择满足大部分项的 prompt。

- 可见症状与真正根因不在同一位置。
- 正确修复很可能同时触及后端逻辑和下游读取方或渲染方。
- 任务包含退化数据或边界状态要求。
- 至少一个关键文件可能在首次 grep 命中范围之外。
- 单文件修复明显是不完整的。
- 仓库中已存在可以指导正确解决方案的模式。
- 验证可以区分狭窄、仔细的修复和嘈杂的补丁。
- 一个合理的浅层修复在公平的共享评分标准下仍会让 qwen 得分低于 `0.7`。
- 更强的端到端修复自然会创造 `>20%` 的相对通过率差距。

如果少于六项为真，则搜索更强的任务。

## 评分标准兼容性要求

假设配对评测者将使用一套共享的 1-5 分 likert 评分标准，包含 5-15 条标准，并试图满足交付约束。

因此 prompt 必须使编写奖励以下方面的评分标准变得容易：

- 根因理解
- 跨文件完整性
- 隐藏边界状态处理
- 退化数据行为
- 验证力度
- 交付质量

并使浅层修复难以看起来"几乎完成"。

如果你无法想象一套公平的共享评分标准可能产生：

- qwen `< 0.7`
- claude `> qwen`
- 相对差距 `> 20%`

那么这个任务还不够强，应该替换。

## 必需的输出行为

除非用户明确要求不同的路径或文件布局，始终执行以下所有操作：

1. 如果 `E:\claudeCoding\dist\<repo-name>` 不存在，则创建它
2. 将最终的 benchmark prompt 写入 `E:\claudeCoding\dist\<repo-name>\prompt.txt`
3. 告知用户 `prompt.txt` 已写入
4. 同时返回简短的人类可读的解释，说明此任务为什么应该能分离 qwen 和 opus

prompt 文件是主要输出。不要在仅于聊天中展示 prompt 后就停止。

在写入 `prompt.txt` 后，除非用户要求更少内容，返回以下章节：

### 2. 为什么此任务能分离模型

- 为什么浅层局部推理很可能失败
- 为什么此任务应有利于 opus 风格的优势

- 解释质量

## 难度启发式

优先选择具有以下属性的 prompt：

- 相关证据分布在多个文件中
- 至少一个需要修改的文件未被明显症状直接点名
- 正确修复需要理解现有抽象
- 验证需要阅读或运行项目特定的检查
- 最佳解决方案是狭窄且精确的，而非宽泛且嘈杂的
- 任务通过留下可检测的下游回归来惩罚单层修复
- 任务支撑一套公平的共享评分标准，其中部分修复自然会丢掉足够多的分以保持在 `0.7` 以下

避免具有以下属性的 prompt：

- 一次 grep 就能立即揭示答案
- 修改主要是机械性的
- 任务可以在不理解架构的情况下解决
- 许多同样草率的解决方案看起来都能通过
- 页面级权限或可见性补丁即使深层一致性仍然被破坏也能捕获大部分分数
- 任务需要人为调整评分标准才能达到期望的差距

## 好的 benchmark 类别

在合适时使用以下类别，按任务类型分组。

### bug修复类别

- 由不一致的边界情况处理导致的隐藏回归
- 验证和运行时逻辑分歧的不完整功能路径
- 破坏了一个集成路径的重构后果
- 掩盖真实生产面正确性问题的测试缺口
- 类型、解析器逻辑和下游使用之间的契约漂移
- 跨多个模块的生命周期或钩子顺序 bug
- 留下过期引用或部分数据的删除 / 同步 / webhook 后果
- 在可空关联下静默丢弃记录的聚合或连接

### 功能优化类别

- 隐藏的 O(n²) 或 N+1 查询，数据量小时不暴露
- 缓存的时机和粒度决策，涉及数据新鲜度与性能的权衡
- 批处理和流式处理之间的转换，涉及内存与延迟的权衡
- 锁粒度或并发模型调整，涉及吞吐量与正确性的权衡
- 内存分配热点，需要理解引用生命周期才能安全优化

### 功能迭代类别

- 多态扩展：在现有接口上新增实现，要求符合 LSP 和现有约定
- 配置驱动扩展：新增一种配置源、格式、协议，融入现有加载管道
- 流水线插拔：在现有 filter/interceptor/pipeline 中插入新的处理节点
- 对称实现：参考 N 个已有同类模块，新增第 N+1 个且保持一致性

### 受限重构类别

- 单体的战略拆分：一个大类拆为职责清晰的小类，公共入口不变
- 隐式依赖显式化：把全局状态、单例、静态方法调用改成通过构造注入
- 条件逻辑多态化：if-else 或 switch 瀑布改成策略模式，行为不变
- 适配器隔离：在外部库和内部代码之间插入适配层，切换实现而不影响调用方

## 最终质量门槛

一个好的 benchmark 任务应该是：

- 对仓库来说是现实合理的
- 不平凡但可解决的
- 很难用浅层推理完美解决
- 很可能产生更清晰的有利于 opus 的得分差距
- 很可能在不需要人为评分技巧的情况下支撑配对交付目标

如果仓库中没有适合此类任务的好机会，指出问题并提出最接近的可行替代方案，而非强行生成一个弱的 benchmark。
