# Repo Difficulty Benchmark Designer

使用此 skill 从 `E:\claudeCoding\repos` 下的仓库中设计 benchmark 任务，用于 qwen 和 opus 之间的对比评测。

## 何时调用

当用户要求以下操作时调用此 skill：

- 从 `E:\claudeCoding\repos` 中的仓库创建任务或 prompt
- 为 qwen 和 opus 设计 benchmark 问题
- 创建旨在拉大两个模型得分差距的高难度评测任务
- 准备后续用于 `prompt.txt` 或评测流程的基于仓库的 prompt

不要将此 skill 用于通用代码解释或普通功能实现。

## 核心目标

你的目标是设计能最大化 opus 在同一仓库任务上得分高于 qwen 的概率的任务。

不要承诺必然的排序结果。而是明确优化以下方面：

- 更高概率让 opus 表现更好
- 更低概率让 qwen 通过浅层局部推理解决好任务
- 最终质量评分中更强的区分度

## 交付目标对齐

此 skill 必须与评测侧的交付约束保持一致。

设计的 prompt 应让合理的配对评测能够满足以下所有条件：

- qwen 通过率 **< 0.7**
- claude 通过率 **> qwen**
- 相对差距 **(claude - qwen) / qwen > 20%**

这意味着任务不能仅仅允许产生一个赢家；它应在共享评分标准下自然地支撑一个**清晰且有证据支撑的差距**。

避免以下类型的任务：

- 浅层部分修复看起来仍然接近完成
- 页面级别的补丁就能获取大部分可用分数
- 两个模型很可能落在同一分数段，仅有微小差异
- 任务过于模糊，导致评分标准后续需要做人为的分离工作

## 语言和语气默认值

除非用户明确要求其他语言或风格：

- 使用**中文**撰写 benchmark prompt
- 以**真实内部工程任务**的语气撰写，而非 AI 指令
- 读起来像技术负责人、代码审查者或同事在实际团队中会发送的内容
- 优先使用**纯文本**，而非 Markdown 重格式
- 保持 prompt **紧凑到可以粘贴进 Claude Code 而不易被截断**

优选以下格式：

- bug 修复任务描述
- 回归问题跟进
- 内部 issue / ticket
- 阻塞发版的修复请求
- 带明确约束的重构清理请求

避免听起来过于 AI 化的 prompt 语言，例如：

- "你正在处理一个..."
- "你的任务是..."
- "请扮演..."
- 过于戏剧化的 benchmark 框架
- 暴露这是一个评测设置的通用元语言

prompt 应感觉像一份真实的开发请求，包含具体的症状、约束和验收标准。

## 输出紧凑性默认值

除非用户明确要求更长的 prompt：

- 优先使用 `prompt.txt` 而非 `prompt.md`
- 尽可能将主 prompt 控制在 **8-20 行**左右
- 使用简短的章节标签，如 `问题` / `要求` / `输出`
- 避免长篇叙述段落
- 仅保留对实现质量有实质影响的约束

如果压缩后任务会变得过于模糊，保持简洁但仍需完整。

## 工作区约定

假设以下布局：

```text
E:\claudeCoding\
  repos\
    repo-1\
    repo-2\
    ...
  dist\
    repo-1\
    repo-2\
    ...
```

从 `E:\claudeCoding\repos\<repo-name>` 读取目标仓库。

将生成的 prompt 写入：

`E:\claudeCoding\dist\<repo-name>\prompt.txt`

如果 `E:\claudeCoding\dist\<repo-name>` 不存在，先创建它。

## 设计理念

倾向于奖励全局推理、架构判断、精确性和有纪律的验证的任务。

任务应让薄弱的局部启发式方法失败。优先选择要求模型完成以下工作的任务：

- 跨多个文件或层级连接行为
- 从现有代码模式中推断隐藏的不变量
- 在修改内部实现时保留架构和公共契约
- 诊断微妙的正确性问题，而非做明显的编辑
- 在多个竞争的实现策略中做出选择并论证选择理由
- 以符合仓库惯例的方式添加或更新验证
- 在代码修改后清晰地解释权衡

避免容易被局部编辑暴力解决的任务，例如：

- 症状明显的孤立单文件 bug 修复
- 搜索替换式的重命名
- 简单的 CRUD 新增
- 有明确指令的直接语法迁移
- 不需要阅读更广泛的仓库上下文就能判断成功的任务

## 分离经验教训

将以下经验作为未来 prompt 设计的硬性指导。

### 1. 为什么早期的权限任务分离不够

权限 / 认证类的 prompt 看似很强，但当以下情况出现时，用浅层推理也能近似解决：

- 症状已经直接指向修复形式，例如"某页面越权可见"或"非创建者还能编辑/删除"
- 明显响应是在指定页面或操作中添加 redirect / guard / 创建者检查
- 大部分重要文件已经被症状本身点名
- 部分修复在评分中仍然看起来可接受，因为页面 guard + 一个服务端检查就覆盖了大部分可见症状
- 几乎没有隐藏的数据契约工作、查询行为、空状态处理或生命周期一致性需要推理

将这类任务视为**中风险分离任务**，而非优先默认任务。

### 2. 为什么后来的已删除用户数据链任务分离更好

当可见症状和真正修复被多层隔开时，出现了更强的任务：

- 外部触发器或生命周期事件
- 清理或变更逻辑
- 数据库引用一致性
- 空引用或悬空引用下的聚合 / 查询行为
- UI 渲染回退
- 未被第一个症状点名的二级页面

这种形态分离更好，因为浅层修复很可能停在一层，而更强的模型必须：

- 在与症状不同的模块中找到非显而易见的根因
- 同时修复数据清理和查询行为
- 在退化数据下保持公开浏览和历史展示稳定
- 推理可为空的运行时状态，而非仅处理正常路径类型
- 在修复后验证不止一条路径

### 3. 未来任务的默认偏好顺序

在选择候选任务时，强烈偏好以下形态的 prompt：

- 删除 / 同步 / webhook / 重试 / 对账之后的数据生命周期回归
- 跨越 schema、查询和渲染层的引用一致性 bug
- 缺失关联导致静默丢数据的聚合或分页 bug
- 运行时数据实际状态与声明的类型 / 契约之间的状态漂移
- 一次修复必须同时保证公开页面、历史页面和管理页面正确的多界面回归

优先于主要归结为以下内容的任务：

- 添加权限检查
- 添加重定向
- 隐藏一个按钮
- 修补一个明显的服务端 action guard
- 在症状直接点名的文件中修改一个条件判断

## 仓库选择工作流

### 1. 阅读仓库上下文

检视目标仓库以理解：

- 架构边界
- 主要包或模块
- 现有测试、脚本和验证机制
- 编码惯例
- 具有跨文件耦合的组件

### 2. 搜索强候选任务区域

优先选择具有以下一项或多项属性的代码区域：

- 多个交互模块
- 有状态的流程或生命周期钩子
- 需要保持一致性的错误处理路径
- 必须保持对齐的配置和运行时行为
- 揭示意图但不揭示确切修复的测试
- 遥测、追踪、缓存、并发、schema 转换、适配器或协议处理
- 可能留下悬空数据的清理 / 删除 / webhook / 同步 / 重试流程
- 在缺失关联时行为不同的聚合、连接、分页、派生列表或回退渲染

### 3. 选择有利于 opus 的任务形态

高价值任务形态包括：

- 根因非显而易见的多文件 bug 诊断
- 必须保留外部行为的受限重构
- 需要推断不变量的部分实现补全
- 添加暴露隐藏正确性问题的验证或测试
- 将多个代码路径对齐到单一架构规则
- 修复证据分布在实现、测试和配置中的 bug
- 数据变更和下游读取方都必须修复的生命周期回归
- 必须在后端和 UI 之间一致处理运行时可空性、过期引用或部分数据的 bug

### 4. 增大分离压力

在撰写任务时，以保持现实感的方式增加难度：

- 要求触及多个相关文件
- 要求保留向后兼容性或接口形态
- 要求从现有模式中推理，而非发明新风格
- 要求验证，而非仅做代码编辑
- 要求简洁解释权衡或残余风险
- 保留一定模糊性让模型自己决策，但不能多到让任务不公平
- 确保至少一个关键文件未被症状明确点名
- 确保浅层单层修复会在查询、渲染、类型或清理行为中留下明显的隐藏缺口
- 确保部分修复在完整性、隐藏边界处理、退化数据行为、验证深度和最终交付质量上自然丢分

## 任务撰写规则

在生成 prompt 时：

- 写成一个真实的工程任务
- 默认使用中文，除非用户明确要求其他语言
- 使其听起来像真实面向开发者的请求，而非 benchmark prompt
- 在 prompt 本身中不要提及 qwen 或 opus
- 不要透露预期的评分标准或隐藏的 benchmark 策略
- 保持足够独立以可执行，但不要过度指定
- 要求真实的代码修改，而非仅分析
- 当仓库允许时要求验证
- 要求最终简短总结改了什么以及如何验证的
- 优先使用压缩的纯文本结构而非长篇散文

一个强 prompt 通常包含：

- 可信的背景或最近的回归上下文
- 具体的用户面或生产面症状
- 与仓库匹配的工程约束
- 隐含着跨文件正确性的验收标准
- 以正常团队请求语气表述的验证期望
- 至少一个迫使解决者保留退化数据行为而不仅修复正常路径的约束
- 至少一个浅层页面级或单文件修复明显无法满足的验收标准

避免：

- 听起来像 benchmark 的开场语言
- 夸张的叙事
- 通用的 AI 助手措辞
- 向解决者过度解释隐藏意图
- 明显最佳响应只是"加权限校验"或"补一个 redirect"的 prompt

## Prompt 压力检查清单

在最终确定任务之前，检查以下所有项。优先选择满足大部分项的 prompt。

- 可见症状与真正根因不在同一位置。
- 正确修复很可能同时触及后端逻辑和下游读取方或渲染方。
- 任务包含退化数据或边界状态要求。
- 至少一个关键文件可能在首次 grep 命中范围之外。
- 单文件修复明显是不完整的。
- 仓库中已存在可以指导正确解决方案的模式。
- 验证可以区分狭窄、仔细的修复和嘈杂的补丁。
- 一个合理的浅层修复在公平的共享评分标准下仍会让 qwen 得分低于 `0.7`。
- 更强的端到端修复自然会创造 `>20%` 的相对通过率差距。

如果少于六项为真，则搜索更强的任务。

## 评分标准兼容性要求

假设配对评测者将使用一套共享的 1-5 分 likert 评分标准，包含 5-15 条标准，并试图满足交付约束。

因此 prompt 必须使编写奖励以下方面的评分标准变得容易：

- 根因理解
- 跨文件完整性
- 隐藏边界状态处理
- 退化数据行为
- 验证力度
- 交付质量

并使浅层修复难以看起来"几乎完成"。

如果你无法想象一套公平的共享评分标准可能产生：

- qwen `< 0.7`
- claude `> qwen`
- 相对差距 `> 20%`

那么这个任务还不够强，应该替换。

## 必需的输出行为

除非用户明确要求不同的路径或文件布局，始终执行以下所有操作：

1. 如果 `E:\claudeCoding\dist\<repo-name>` 不存在，则创建它
2. 将最终的 benchmark prompt 写入 `E:\claudeCoding\dist\<repo-name>\prompt.txt`
3. 告知用户 `prompt.txt` 已写入
4. 同时返回简短的人类可读的解释，说明此任务为什么应该能分离 qwen 和 opus

prompt 文件是主要输出。不要在仅于聊天中展示 prompt 后就停止。

## 推荐的响应章节

在写入 `prompt.txt` 后，除非用户要求更少内容，返回以下章节：

### 1. Prompt 已保存

写明确切的输出路径。

### 2. 为什么此任务能分离模型

简要解释：

- 哪些仓库特征使任务变难
- 为什么浅层局部推理很可能失败
- 为什么此任务应有利于 opus 风格的优势
- 较弱修复可能遗漏哪个隐藏层或二级界面
- 为什么此任务很可能支撑 `qwen < 0.7`、`claude > qwen` 和 `diff > 20%`

### 3. 预期的高层解决方案形态

描述一个强解决方案大概会包含什么，而不给出完整补丁。

### 4. 评测关注点

列出最可能区分两个模型的维度，例如：

- 在仓库约束下的正确性
- 跨文件一致性
- 验证质量
- 克制和范围控制
- 解释质量
- 生命周期和退化数据处理

## 难度启发式

优先选择具有以下属性的 prompt：

- 相关证据分布在多个文件中
- 至少一个需要修改的文件未被明显症状直接点名
- 正确修复需要理解现有抽象
- 验证需要阅读或运行项目特定的检查
- 最佳解决方案是狭窄且精确的，而非宽泛且嘈杂的
- 任务通过留下可检测的下游回归来惩罚单层修复
- 任务支撑一套公平的共享评分标准，其中部分修复自然会丢掉足够多的分以保持在 `0.7` 以下

避免具有以下属性的 prompt：

- 一次 grep 就能立即揭示答案
- 修改主要是机械性的
- 任务可以在不理解架构的情况下解决
- 许多同样草率的解决方案看起来都能通过
- 页面级权限或可见性补丁即使深层一致性仍然被破坏也能捕获大部分分数
- 任务需要人为调整评分标准才能达到期望的差距

## 好的 benchmark 类别

在合适时使用以下类别：

- 由不一致的边界情况处理导致的隐藏回归
- 验证和运行时逻辑分歧的不完整功能路径
- 破坏了一个集成路径的重构后果
- 掩盖真实生产面正确性问题的测试缺口
- 类型、解析器逻辑和下游使用之间的契约漂移
- 跨多个模块的生命周期或钩子顺序 bug
- 留下过期引用或部分数据的删除 / 同步 / webhook 后果
- 在可空关联下静默丢弃记录的聚合或连接

## 最终质量门槛

一个好的 benchmark 任务应该是：

- 对仓库来说是现实合理的
- 不平凡但可解决的
- 很难用浅层推理完美解决
- 很可能产生更清晰的有利于 opus 的得分差距
- 很可能在不需要人为评分技巧的情况下支撑配对交付目标

如果仓库中没有适合此类任务的好机会，指出问题并提出最接近的可行替代方案，而非强行生成一个弱的 benchmark。
