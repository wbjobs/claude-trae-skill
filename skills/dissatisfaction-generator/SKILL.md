# 不满意原因生成器

## 使用说明

1. 用户指定 output 目录下的某个 txt 文件名（如 `round1.txt`）
2. 读取该文件内容，分析其中的代码生成结果
3. 在 product 目录下生成同名文件
4. 文件内容为按照以下规范生成的「产物不满意」和「过程不满意」描述

## 任务说明

对 AI Agent 的多轮代码生成结果进行满意度标注。每一轮输出包含「产物不满意」和「过程不满意」两部分。严格按照以下规范执行，不得遗漏或违反。

## 基础结构要求

有几轮就分几轮，每轮格式为：

```text
# 第N轮
产物不满意：XXXXX
过程不满意：XXXXX
```

全程使用自然白话、委婉语气，表述方向统一为「效果不佳、优化不到位、功能不完善、存在明显偏差、没能有效改善」。

**禁止使用极端绝对词**（完全没实现、彻底失效、全部崩溃、全部未达成等）。

**禁止使用笼统短句**（无法运行、直接报错、问题没修复、接口失败等）。

**禁止使用生硬 AI 模板话术**。

## 产物不满意强制内容

每一条「产物不满意」描述必须至少包含以下 2 类要素（可融合在一段话中），并且明确指出问题所在的类名、方法名、函数名或具体逻辑模块：

1. **范围/对象**：出问题的类、方法、函数、计算逻辑、拓展功能组件（必须写出具体名称）。
2. **现象/证据**：实际运行表现、数值偏差、输出错乱、异常行为等客观现象。
3. **与需求的偏差**：对比本轮开发需求，实际缺失的能力、优化效果不足、未达预期目标。
4. **影响范围**：干扰的业务流程、数据分析流程、模型评估流程、可视化展示等。
5. **复现条件**：触发问题的数据集、输入参数、样本规模、特殊数据分布等场景。
6. **严重程度**：边界场景轻微异常、部分场景优化失效、拓展功能完善度不足、评估参考效果差。

### 示例

attendance_service.py 中的 AttendanceDetectionService._match_punches_to_shift 方法，在处理跨天班次时未将下班时间顺延至次日，导致次日凌晨的下班打卡无法被正确关联，跨天班次的早退与缺卡判断完全错误。对比需求（通用考勤异常检测），夜班场景功能不完善，影响所有跨天班次，属于部分场景优化失效。

**禁止**：产物不满意中出现对过程的评价（如"本轮、第几轮、提示词、日志、轨迹、bug修复、功能迭代"等敏感词），也禁止提及模型本身的工具或方法。

## 过程不满意强制内容

过程不满意需要判断模型是否：

- 充分理解用户需求（目标是否明确）
- 推理过程是否精简合理
- 任务规划是否合理
- 操作过程是否合理，是否存在高危操作或缺乏边界感的行为
- 总结是否正确，有无虚假完成情况

### 硬性要求

必须结合本轮实际运行情况（但不出现"日志轨迹"等敏感词），定位到具体环节或步骤（例如："在编写匹配函数之前""在规划任务清单后""在修改时间计算逻辑时"等）。

使用证据，不用形容词。例如：指出遗漏了哪个边界条件、漏掉了哪条需求、哪个文件中的哪段逻辑存在缺陷。

只能写由模型自身能力造成的问题，不写环境或网络波动导致的问题。

**一条合格的过程不满意必须包含三要素**：

1. **触发节点（When）**：在哪个阶段或哪个动作出了问题？
2. **实际行为（What）**：模型做错了什么？
3. **业务影响（Impact）**：这个错误导致了什么后果？

鼓励补充根因和正确做法。

不能只写历史遗留问题，必须本轮评价为主（可以附加历史问题，但不能颠倒或缺前者）。

### 格式参考

在编写 _match_punches_to_shift 之前，未主动识别班次时间可能跨天的业务场景。在创建任务清单后，直接假设上下班在同一天并编写了匹配逻辑，没有增加边界条件判断。这导致生成的产物功能不完善，需在后续轮次返工修复。根因在于需求理解不够深入，正确做法应先列举所有可能的班次类型，再设计通用的时间对齐函数。

**禁止**：出现模型本身使用的工具名称（如 Write、edit_file_search_replace、run_command、todo_write 等）、禁止出现"模型"二字（必要时可用"agent"或"生成结果"替代），禁止出现"日志轨迹""提示词"等敏感词。

## 输出示例（完整轮次）

```text
# 第1轮
产物不满意：`attendance_service.py` 中的 `AttendanceDetectionService._match_punches_to_shift` 方法，默认上班日期与下班日期相同。当班次为跨天模式（22:00-06:00）时，未将下班时间顺延至次日，导致次日凌晨打卡无法关联，跨天班次的早退与缺卡判断完全错误。对比需求（通用考勤检测），夜班场景功能不完善，影响所有跨天班次，复现条件为结束时间≤开始时间且打卡跨天，属于部分场景优化失效。

过程不满意：在编写匹配函数前，未识别跨天班次场景。创建任务后直接假设同一天，未增加边界判断，导致产物需返工。根因在于需求理解不深入，正确做法应先梳理班次类型再设计通用函数。

# 第2轮
产物不满意：`attendance_service.py` 修复后，`_get_scheduled_end` 和 `detect_exceptions` 中日期偏移逻辑重复出现（如 `detection_date + timedelta(days=1)` 在多处独立书写），未抽取公共方法。虽然功能满足，但代码复用性不足，增加了维护成本，属于拓展功能完善度上的轻微不足。

过程不满意：分别修改了五个独立函数，但未同步抽取公共的日期偏移工具函数。若能在第一次修改时提炼辅助方法，可减少重复代码。根因在于任务规划时未考虑代码复用，正确做法应先设计统一的跨天日期计算工具。

# 第3轮
产物不满意：`attendance_service.py` 中的 `_match_punches_to_shift` 方法，在整合补签记录时未对补签类型赋予最高匹配优先级。当员工已有真实打卡（09:15）且补签上班卡（08:52）存在时，真实打卡仍被当作上班卡，导致补签后出现错误异常（如早退515分钟）。对比需求（审批后自动修正考勤），该缺陷使补签引入新异常。复现条件为补签时间早于真实打卡且类型相同，影响所有替换场景，属于部分场景优化失效。

过程不满意：实现核心方法后，未针对"既有真实打卡又有补签打卡"的混合场景编写专门验证用例。导致示例脚本运行时才暴露问题，需再次修改优先级逻辑。根因在于测试覆盖不充分，正确做法应先列出所有补签场景并逐一验证。
```

## 最后强调

1. **产物不满意中必须写出具体的类名、方法名、函数名或逻辑模块**。
2. **过程不满意中不得出现任何工具名称、方法名称、敏感词**（如 run_command、Write、edit_file_search_replace、todo_write、日志轨迹等）。
3. 全程使用自然白话，语气委婉，表述方向统一为"效果不佳、优化不到位、功能不完善、存在明显偏差、没能有效改善"。
4. 严格遵循三要素和至少2类要素的要求。
