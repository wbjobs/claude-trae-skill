# Gene

## 适用场景

当用户要求你对 `data.xlsx` 逐行生成不满意原因，并将结果写回工作簿时调用本技能。

## 目标

1. 读取 `data.xlsx` 中每一行的 `user prompt` 列与 `日志轨迹` 列。
2. 根据这两列内容生成「产物不满意」与「过程不满意」。
3. 将生成结果写入对应行的 `不满意原因` 列。
4. 如果某行的 `user prompt` 或 `日志轨迹` 为空，则跳过该行，不生成内容。

## 执行要求

- 先确认工作区存在 `data.xlsx`。
- 读取工作簿表头，优先按精确列名匹配：`user prompt`、`日志轨迹`、`不满意原因`。
- 逐行处理数据，只改写目标列，不破坏其他列内容与顺序。
- 生成内容时，输出必须严格为两行：

产物不满意：XXXXX
过程不满意：XXXXX

- 若已有 `不满意原因` 内容，只有在用户明确要求覆盖时才覆盖；若用户未说明，默认直接回填当前生成结果。
- 写回后应保存到原始 `data.xlsx`。

## 生成规范

### 一、任务说明

你需要对 AI Agent 的提示词和日志轨迹进行满意度标注。输出包含「产物不满意」和「过程不满意」两部分。严格按照以下规范执行，不得遗漏或违反。

### 二、基础结构要求

格式为：

产物不满意：XXXXX
过程不满意：XXXXX

全程使用自然白话、委婉语气，表述方向统一为「效果不佳、优化不到位、功能不完善、存在明显偏差、没能有效改善」。

禁止使用极端绝对词（完全没实现、彻底失效、全部崩溃、全部未达成等）。

禁止使用笼统短句（无法运行、直接报错、问题没修复、接口失败等）。

禁止使用生硬 AI 模板话术。

### 三、产物不满意强制内容

每一条「产物不满意」描述必须至少包含以下 2 类要素（可融合在一段话中），并且明确指出问题所在的类名、方法名、函数名或具体逻辑模块：

- 范围/对象：出问题的类、方法、函数、计算逻辑、拓展功能组件（必须写出具体名称）。
- 现象/证据：实际运行表现、数值偏差、输出错乱、异常行为等客观现象。
- 与需求的偏差：对比本轮开发需求，实际缺失的能力、优化效果不足、未达预期目标。
- 影响范围：干扰的业务流程、数据分析流程、模型评估流程、可视化展示等。
- 复现条件：触发问题的数据集、输入参数、样本规模、特殊数据分布等场景。
- 严重程度：边界场景轻微异常、部分场景优化失效、拓展功能完善度不足、评估参考效果差。

示例：

attendance_service.py 中的 AttendanceDetectionService._match_punches_to_shift 方法，在处理跨天班次时未将下班时间顺延至次日，导致次日凌晨的下班打卡无法被正确关联，跨天班次的早退与缺卡判断完全错误。对比需求（通用考勤异常检测），夜班场景功能不完善，影响所有跨天班次，属于部分场景优化失效。

禁止：产物不满意中出现对过程的评价（如“本轮、第几轮、提示词、日志、轨迹、bug修复、功能迭代”等敏感词），也禁止提及模型本身的工具或方法。

### 四、过程不满意强制内容

过程不满意需要判断 agent 是否：

- 充分理解用户需求（目标是否明确）
- 推理过程是否精简合理
- 任务规划是否合理
- 操作过程是否合理，是否存在高危操作或缺乏边界感的行为
- 总结是否正确，有无虚假完成情况

硬性要求：

- 必须结合实际执行记录中的运行情况，但不出现“日志轨迹”等敏感词。
- 必须定位到具体环节或步骤，例如“在编写匹配函数之前”“在规划任务清单后”“在修改时间计算逻辑时”。
- 使用证据，不用空泛形容词。要指出遗漏了哪个边界条件、漏掉了哪条需求、哪个文件中的哪段逻辑存在缺陷。
- 只能写由 agent 自身能力造成的问题，不写环境或网络波动导致的问题。

一条合格的过程不满意必须包含三要素：

- 触发节点（When）：在哪个阶段或哪个动作出了问题。
- 实际行为（What）：agent 做错了什么。
- 业务影响（Impact）：这个错误导致了什么后果。

鼓励补充根因和正确做法。

不能只写历史遗留问题，必须以本轮评价为主（可以附加历史问题，但不能颠倒或缺前者）。

格式参考：

在编写 _match_punches_to_shift 之前，未主动识别班次时间可能跨天的业务场景。在创建任务清单后，直接假设上下班在同一天并编写了匹配逻辑，没有增加边界条件判断。这导致生成的产物功能不完善，需在后续轮次返工修复。根因在于需求理解不够深入，正确做法应先列举所有可能的班次类型，再设计通用的时间对齐函数。

禁止：出现工具名称、禁止出现“模型”二字，必要时可用“agent”或“生成结果”替代；禁止出现“日志轨迹”“提示词”等敏感词。

### 五、输出示例

产物不满意：`attendance_service.py` 中的 `AttendanceDetectionService._match_punches_to_shift` 方法，默认上班日期与下班日期相同。当班次为跨天模式（22:00-06:00）时，未将下班时间顺延至次日，导致次日凌晨打卡无法关联，跨天班次的早退与缺卡判断完全错误。对比需求（通用考勤检测），夜班场景功能不完善，影响所有跨天班次，复现条件为结束时间≤开始时间且打卡跨天，属于部分场景优化失效。

过程不满意：在编写匹配函数前，未识别跨天班次场景。创建任务后直接假设同一天，未增加边界判断，导致产物需返工。根因在于需求理解不深入，正确做法应先梳理班次类型再设计通用函数。

### 六、最后强调

- 产物不满意中必须写出具体的类名、方法名、函数名或逻辑模块。
- 过程不满意中不得出现任何工具名称、方法名称、敏感词（如 run_command、Write、edit_file_search_replace、todo_write、日志轨迹等）。
- 全程使用自然白话，语气委婉，表述方向统一为“效果不佳、优化不到位、功能不完善、存在明显偏差、没能有效改善”。
- 严格遵循三要素和至少 2 类要素的要求。
- 如果 `user prompt` 或 `日志轨迹` 为空值，则跳过该行，不生成不满意原因。

## 建议执行流程

1. 读取 `data.xlsx`，定位三列。
2. 遍历每一行，检查 `user prompt` 与 `日志轨迹` 是否同时非空。
3. 对满足条件的行生成两段式结果。
4. 将结果写入 `不满意原因` 列。
5. 保存原文件。
6. 完成后汇报处理行数、跳过行数，以及是否存在缺失列名的情况。

## 质量检查

在结束前自行复核以下内容：

- 每条结果是否严格为两行。
- 「产物不满意」是否写出具体类名、方法名、函数名或逻辑模块。
- 「过程不满意」是否包含触发节点、实际行为、业务影响。
- 是否避免了敏感词、工具名、极端绝对词、笼统短句。
- 是否跳过了空白行。
- 是否已将结果写回原始文件。
